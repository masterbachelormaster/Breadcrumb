🟡 **Breadcrumb/Services/OpenRouterProvider.swift:119-124**
`extractJSONObject` in OpenRouterProvider.swift:119-124 uses `firstIndex(of: "{")` + `lastIndex(of: "}")` to locate the JSON payload. If a provider ignores the `response_format: json_object` directive and emits trailing prose containing a `}` (e.g. `…I hope this helps :}`), the slice extends past the real JSON and decoding fails with `invalidResponse`. A brace-balanced scanner or `JSONSerialization`-based prefix consumer would be more robust, though the failure mode is graceful (a surfaced error, not silent corruption).

🔴 **Breadcrumb/Views/DocumentListView.swift:141-144**
**Stale-bookmark refresh can silently destroy a valid `LinkedDocument`.** In `DocumentListView.openDocument` the `if isStale` branch writes `doc.bookmarkData = try? url.bookmarkData()` unconditionally and persists. If `url.bookmarkData()` throws (transient I/O, network volume briefly unmounted, sandbox/permission blip), `try?` yields `nil` and the previously-valid bookmark is overwritten with `nil` and saved — permanently breaking the link with no in-app recovery. Fix: `if let newData = try? url.bookmarkData() { doc.bookmarkData = newData; modelContext.saveWithLogging() }`.

🟡 **Breadcrumb/Views/BulletableField.swift:63-72**
In `BulletableField.swift:66-69`, pressing Enter on an empty bullet row keeps appending more empty rows with no upper bound — `insertBullet(after:)` never checks whether the current row is empty. There is no keyboard path to exit list mode (only the minus button drops back to plain mode, and only when count falls below 2), so rapid Enter presses produce a growing column of empty bullets that the user must dismiss one by one. The same issue affects `addBullet()` on repeated Add-bullet taps. Consider adopting the convention used by Apple Notes, Reminders, and Notion: Enter on an empty trailing bullet exits list mode instead of inserting another empty row.

🔴 **Breadcrumb/Views/PomodoroRunningView.swift:144-162**
The 'Stop Completely' button at break-end (both cycle-complete and regular paths in PomodoroSessionEndView.swift:61-82) calls onStopCompletely, whose handler in PomodoroRunningView.swift:144-162 unconditionally inserts a new `.work` PomodoroSession — duplicating the session that was already saved moments earlier by 'Save and Break' / 'Skip'. This silently inflates Project.completedPomodoroCount and totalFocusTime in stats and rollups. Fix by branching the handler on `wasBreakEnd` (or splitting into separate callbacks) so the break-end path only stops the timer instead of persisting another work session.

🔴 **Breadcrumb/Views/PomodoroSessionEndView.swift:182-184**
`PomodoroSessionEndView.saveAndBreak` (lines 182-184) and `PomodoroRunningView.onSkip` (PomodoroRunningView.swift:129-132) both hardcode `session.completed = true` and compute `actualDuration = originalDurationSeconds + overtimeSeconds`. But both paths are reachable mid-session via `stopSession()` (PomodoroRunningView.swift:210-214), which just pauses the timer without waiting for `.sessionEnded` — so stopping a 25-min Pomodoro after 10 minutes and clicking "Save & Break" or "Skip" records a completed session with `actualDuration=1500s`, inflating `Project.completedPomodoroCount` and `Project.totalFocusTime`. The sibling `onStopCompletely` handler (lines 151-153) uses the correct formula `completed = remainingSeconds <= 0` and `actualDuration = originalDurationSeconds - remainingSeconds + overtimeSeconds` — apply that same formula to both sites.

🔴 **Breadcrumb/Views/PomodoroSessionEndView.swift:212**
The FocusMate "Save & Done" path in PomodoroSessionEndView.saveAndDone (line 212) computes actualDuration as originalDurationSeconds - remainingSeconds, but for FocusMate sessions originalDurationSeconds is the full planned meeting length while phaseDurationSeconds reflects the actual (potentially shorter) in-session time when the user joins a meeting via a past start time. When a user joins a 50-minute meeting 10 minutes late and lets the timer naturally expire, the session records 50 minutes of focus time instead of ~40, inflating Project.totalFocusTime. Fix by using phaseDurationSeconds - remainingSeconds (or elapsed time from phaseStartDate).

🟡 **Breadcrumb/Models/PomodoroSession.swift:4-20**
